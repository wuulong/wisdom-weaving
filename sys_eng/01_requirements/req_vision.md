# 01. 產品願景與核心需求定義 (Requirements & Vision)

本文件定義專案 **`Wisdom Weaving`** 的核心願景、目標使用者、核心痛點以及系統需求。這是一切規格與設計的起點。

---

## 1. 產品願景 (Product Vision)

給定一本書、或是一個戲劇內容，藉由 Agents 間的提問與歸納，產製知識體系，並藉由整理過的資料與持續建構中的知識體系，能夠深入與系統性的回答使用者問題，並能夠整體性的說明這個知識體系。本專案未來將以**開源/公開釋出**為導向，提供社群作為通用的知識系統建構基礎。

在 AI 時代，想法到實踐的距離 (TTI) 已縮短至接近零，但隨著系統規模與跨領域概念（如宮廷權謀、地緣政治、人情世故）的擴張，傳統開發所面臨的「細節摩擦」與「大模型幻覺（說溜嘴、行為失控）」已成為主要痛點。

知識體系的建構本質上是一個**持續演進與按需動態建置（Just-In-Time）**的過程，而非一次性建立整個完整的體系。因為知識體系的面相與深度，實際取決於使用者的探索需求。我們**只在使用者即將要知道的時候，先行準備與動態建構好**。

* **核心宣言**：為了有效萃取出給定書籍與文件的知識體系，我們打造了**通用 Wisdom Weaving 智慧工程沙盒**。藉由預先整理成 DB, vector DB, 以及 knowledge graph 為基底，利用各司其職的 Agent，藉由反思知識體系的缺漏，持續分析並上網查詢理論基礎，建構知識體系的多維度向量空間，並建構對應目錄內檔案。這是個通用性的方法論，本次以《鹿鼎記》為範例，用以建構與驗證這個方法論。也提供成為下一個知識系統的基礎。

## 2. 目標使用者與關鍵痛點 (Target Persona & Pain Points)

| 使用者角色 (Persona) | 典型使用場景 (Scenario) | 核心痛點 (Pain Points) | 預期獲得的價值 (Value) |
| :--- | :--- | :--- | :--- |
| **AI-Native 總架構師** | 設計跨領域、多重利益衝突的複雜 MAS 系統時。 | 大模型在多重身分下容易「穿幫」或說溜嘴，且無法進行剛性安全與道德阻斷。 | 提供可驗證、可重製的 Hooks 攔截機制與圖譜語意層底座，將控制邏輯與 LLM 解耦。 |
| **智慧工程實驗者** | 如何有效地建構出文本的知識體系，以及檢驗知識體系的完整度與擴充知識體系的內容。 | 手動提取與驗證知識的成本極高，難以評估知識圖譜的涵蓋率與一致性。 | 快速建構、自動化稽核與增量擴充文本知識體系。 |
| **智慧運用者** | 去詢問了解在這個知識體系中自己所需要了解的部分。 | 沒有架構性的框架依循，所以只能亂問，難以獲得系統性的全貌理解。 | 能依循結構化的知識框架進行系統性探索，精準獲取解答。 |

## 3. 使用者故事與核心需求列表 (User Stories & Requirements)

* **`[REQ-001]` 傳統與語意複合式資料庫建置需求**
  * **使用者故事**：身為 系統工程師，我想要 建立傳統關聯式資料庫（同時具備 Vector DB 能力，可以使用語意查詢），並搭配以 JSON 為內容的 Metadata 結構來建立充足的關聯性（如章節存取、Entity 被何處參考、章節摘要等），並融合語意索引與取得機制，以便於 系統既能進行精準的結構化關係查詢，又能進行彈性的語意檢索。
  * **優先級**：`High`
  * **驗收標準 (Acceptance Criteria)**：
    * 1. 支援傳統資料庫儲存章節與實體引用關係，並以 JSON 欄位儲存豐富的 Metadata。
    * 2. 支援查詢特定章節的摘要，以及特定實體（Entity）被哪些章節/段落參考。
    * 3. 資料庫原生支援 Vector Embeddings 儲存與語意相似度檢索（即具備 Vector DB 語意查詢能力）。
    * 4. 系統可針對不同來源的文本，動態建立與初始化對應的獨立資料庫檔案，實現多文本庫的空間隔離與單一庫固化。
  * **追溯規格**：`[追溯規格：SPC-003, SPC-004]`

* **`[REQ-002]` Entity-Relation (ER) 圖譜語意層需求**
  * **使用者故事**：身為 系統工程師，我想要 將小說中的實體（角色、組織、身分、道具）及其關係建立圖譜，以便於 提供 Agent 進行高維度、確定性的地緣政治與人際網絡查詢。
  * **優先級**：`High`
  * **驗收標準 (Acceptance Criteria)**：
    * 1. 支援查詢實體之間的多層關係（如陳近南對韋小寶的恩情值）。
    * 2. 輸出格式必須是結構化數據，便於後續 Agent 讀取。
  * **追溯規格**：`[追溯規格：SPC-005]`

* **`[REQ-003]` 不同 Agent 間之提問回答與知識萃取拓撲結構需求**
  * **使用者故事**：身為 總架構師，我想要 在不同 Agent 間的提問與回答過程中，自動萃取出知識體系，並建立負責回答的 Agent 拓樸架構，以便於 避免單一 Agent 在面對複雜大文本情境下發生 Context 內爆與角色混亂，實現高效且高可讀性的知識體系建構。
  * **優先級**：`High`
  * **驗收標準 (Acceptance Criteria)**：
    * 1. 系統必須定義不同提問與歸納 Agent 間的對話流與協定。
    * 2. 負責回答的 Agent 必須依循已定義的拓本架構（如層級問答或對抗生成）進行資料分發與回答。
    * 3. 萃取之知識體系格式需符合結構化定義。
  * **追溯規格**：`[追溯規格：SPC-001, SPC-002]`

* **`[REQ-004]` 知識缺漏反思與外部理論聯網查詢**
  * **使用者故事**：身為 智慧工程實驗者，我想要 讓 Agent 能主動反思當前建構之知識體系的缺漏與盲點，以便於 自動發起聯網查詢（如 Google Search）以補齊背景理論基礎。
  * **優先級**：`Medium`
  * **驗收標準 (Acceptance Criteria)**：
    * 1. 能夠產出知識缺漏分析報告（Gap Analysis）。
    * 2. 支援聯網查詢並擷取相關參考文獻與理論基礎。
  * **追溯規格**：`[追溯規格：SPC-006]`

* **`[REQ-005]` 多維度向量空間建置與目錄結構檔案產製**
  * **使用者故事**：身為 系統工程師，我想要 根據整理過的資料與外部理論，建構知識體系的多維度向量空間，並自動在對應目錄下產製結構化 Markdown 檔案，以便於 形成可持續演進的知識庫。
  * **優先級**：`High`
  * **驗收標準 (Acceptance Criteria)**：
    * 1. 支援將萃取出的知識體系以層級化目錄與檔案寫入本地端。
    * 2. 生成之檔案格式需符合 markdown 規範，便於語意檢索與持續厚化。
  * **追溯規格**：`[追溯規格：SPC-007]`

* **`[REQ-006]` 系統起始建置與持續運行建構 Skill 需求**
  * **使用者故事**：身為 智慧工程實驗者，我想要 產生一組 Skill，當一個文字文本被提供後，可用於起始建構整個系統（包括資料庫初始化、配接 Agent 拓本），並能在使用者發起探索時，持續按需（JIT）運行以建立與擴展知識體系。
  * **優先級**：`High`
  * **驗收標準 (Acceptance Criteria)**：
    * 1. 當使用者提供一個新文本時，此組 Skill 能自動觸發整個系統的初始化配置。
    * 2. 支援在使用者提出詢問（即將要知道）時，觸發 Agent 動態建構、更新並擴展相關知識。
    * 3. 可藉由 Skill 所提供的指令進行系統控制。
  * **追溯規格**：`[追溯規格：SPC-008]`

* **`[REQ-007]` 整體性自動化腳本、手冊與入口級 CLI 需求**
  * **使用者故事**：身為 系統工程師，我想要 盡量開發具備整體性的 Python 腳本（Scripts）來處理數據與業務邏輯，以提升效率並減少 Token 消耗，並為這些腳本提供完整的手冊說明書以及一個統一的入口級 CLI 介面，以便於 使用者能簡單、高效且精準地操作整個系統。
  * **優先級**：`High`
  * **驗收標準 (Acceptance Criteria)**：
    * 1. 大部分重複性的數據處理、檢索與分析功能皆由 Python 腳本執行，減少 LLM 的冗餘開銷。
    * 2. 核心腳本需配備對應的 Markdown 操作說明手冊。
    * 3. 提供一個統一入口 CLI 指令供使用者與系統進行高效對話與操作。
    * 4. 所有的指令（包括 init、query、restore）與底層運作腳本，均支援傳入指定資料庫的參數（如 `-d` 或 `--db`），以在指定的資料庫範疇下運行。
  * **追溯規格**：`[追溯規格：SPC-009]`

* **`[REQ-008]` 架構性 POC 驗證與輕量核心演進開發原則**
  * **使用者故事**：身為 智慧工程實驗者 / 系統工程師，我想要 先行建立系統的架構性 POC（Proof of Concept）並維護一個輕量核心，以便於 當未來有更多實驗需求時，能先在此輕量核心上進行改善與疊代，隨後再逐步擴展至整體系統。
  * **優先級**：`High`
  * **驗收標準 (Acceptance Criteria)**：
    * 1. 系統開發初期優先實作架構性 POC 核心，驗證提問與萃取鏈路之通順性。
    * 2. 維護清晰的輕量核心（Lightweight Core）架構，避免代碼結構過早膨脹或過度設計。
    * 3. 任何新增的實驗性功能，均需遵循「在核心上疊代優化 $\rightarrow$ 評估成熟 $\rightarrow$ 擴展至整體」的開發原則。
  * **追溯規格**：`[追溯規格：SPC-010]`

* **`[REQ-009]` 無版權模擬文本生成與設計需求**
  * **使用者故事**：身為 智慧工程實驗者，我想要 在 POC 階段設計並自動生成一組適合進行多代理人權謀問答與知識萃取的「無版權模擬文本」，以便於 系統能在開源公開釋出時進行合規的示範與驗證。
  * **優先級**：`High`
  * **驗收標準 (Acceptance Criteria)**：
    * 1. 模擬文本內容必須包含豐富的多重身份、衝突動機與關係拓本，模擬清代宮廷或地緣政治的複雜局勢。
    * 2. 文本本身無 any 著作權/版權爭議，可自由公開釋出。
    * 3. 先用 鹿鼎記摘要 做 POC 示範與驗證。這個可以直接生成，而且無版權疑慮。等這個跑順了，再把整本小說拿進來測。等測試有個基本運行，拿真實鹿鼎記來覆蓋也不違和。
  * **追溯規格**：`[追溯規格：SPC-011]`

* **`[REQ-010]` 鹿鼎記範例去版權化與 L1 文本剝離釋出需求**
  * **使用者故事**：身為 智慧工程實驗者，我想要 在公開釋出本專案時，不提供《鹿鼎記》的版權原始文本，且自動將 SQLite 資料庫中的 Layer 1 原始文本內容 (contents.raw_text) 進行剝離，以便於 在釋出時既能展示建模成果（保留實體、關係與知識中樞），又能避免著作權版權糾紛。
  * **優先級**：`High`
  * **驗收標準 (Acceptance Criteria)**：
    * 1. 提供去版權化（L1 文本剝離）的工具或發佈流程，能將 `contents` 表中的 `raw_text` 內容抹除，但保留對應的 ids、embeddings（向量）與關係結構。
    * 2. 提供本地端文本對齊與還原的說明指引，引導使用者提供本地端原始小說以重新加載 L1 文字。
  * **追溯規格**：`[追溯規格：SPC-012]`

* **`[REQ-011]` Agentic AI 實作 POC 之 antigravity SDK 技術約束需求**
  * **使用者故事**：身為 系統架構師，我想要 在進行 Agentic AI 代理人的實作 POC 時，明確限定選用 antigravity SDK 進行開發與生命週期管理，以便於 系統具備與 IDE/CLI 及 Google 大模型生態系統的高度相容性與剛性攔截控制。
  * **優先級**：`High`
  * **驗收標準 (Acceptance Criteria)**：
    * 1. 代理人 (Agent) 網絡的生命週期、工具調用與對抗鏈路實作，必須呼叫 `google-antigravity` SDK 之 API。
    * 2. POC 開發框架必須完全依循 antigravity 2.0 所定義的代理人設計模式。
  * **追溯規格**：`[追溯規格：SPC-001, SPC-002]`
